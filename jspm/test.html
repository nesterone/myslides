Jargon

  • Package Manager - fetch scripts from some repository, taking dependencies into account.
  • Build Tool - combine scripts and other assets together into something usable by browsers.
  • Register - register of available packages
  • Repository - place to store packages
  • Module System - A way to programmatically load scripts when they’re needed
  • Semantic Versioning (semver) - aka standart convention in nodejs community
  • Asynchronous Module Definition (AMD) - specifies a mechanism for defining modules such that the module and its dependencies can be asynchronously loaded
  • CommonJS (CJS) -
  • ES6 Modules (es6) - upcoming standard for module definition
  • Shim - javascript files which should be wrapped in some boilerplate code to support your module type
  • Assets - resources of your module/package in general, for example: css, images, fonts, javascript etc.

Intro

There is still no best answer to client-side package managers and dependency management.

We still have to cobble together the solutions that work best in the context of each project,

but at least we’re not downloading js/css manually through the browser,

or using wget, or writing bash scripts to download/update our client-side dependencies.

Before we start to look on current state of package management in javascript community let's separate

server from client, because requirement are different

Client requirements

  1. Small size of application
    • Nested dependency tree - big size, not a problem for server, simplify resolving same dependencies
    • Flat dependency tree - small size, good for client, puts the burden of dependency resolution on the use
  2. Client packages need to be able to share css, fonts, images etc.
  3. Clients often need to be able to load additional scripts for application by demand

Package Managers

Let's look on package managers one by one in chronological order.

npm

Npm is the package manager that comes preinstalled with the Node.js server platform.

It is used to install node programs from the npm registry. In short, it is a tool that aids a developer as they install and manager third party node programs.

Don't confuse npm with the commonJS require() statement. It is not used to load code, it is used to install code and manage code dependencies from the command line

Overview

Install dependencies

In project scope

npm install backbone --save

and later in code

var backbone = require('backbone') console.log(backbone.VERSION)

In global scope

npm install jshint -g

So you would have a nice to play with until to check your quality of any js file

jshint --show-non-errors main.js

Package descriptor

Rules:

  • [a] a folder containing a program described by a package.json file
  • [b] a gzipped tarball containing [a]
  • [c] a url that resolves to [b]
  • [d] a <name>@<version> that is published on the registry with [c]
  • [e] a <name>@<tag> that points to [d]
  • [f] a <name> that has a "latest" tag satisfying [e]
  • [g] a git url that, when cloned, results in [a]

Bundled with r.js optimizer, for node modules it uses amdefine to convert from CJS to AMD

Ender

"the no-library library"

It's npm's little sister. Provide piggyback solution on top of npm package manager.

It's hard to say that ender is a 100% package manager, it's more a build tool. Let's name it as build tool on top of npm.

Ender can load any npm packages but has it's own reserved keyword 'ender' to form some sort of ender specific register


Overview

Search package

ender search backbone

Install dependencies

ender add domready

and later in code 

$.domReady(function(){ alert('DOM ready'); });

include ender.js or ender.min.js to your html file.

To make your API accessible from $ object (The Ender Client API) you need to wrap it in bridge boilerplate

Ender doesn't add dependencies to package.json, you need to take care of it.

Also, I've not found any commands which would install all dependencies, so you need to run

npm install

'We put the browser first'

Component

Component is one of many frontend solutions. One of the major differences between Component and other solutions is that it is vertically integrated,

meaning it does everything from package management to building. Of course, to do so, it is opinionated and is not suitable for every workflow.

  • UNIX philosophy

var backbone = require('backbone'); console.log(backbone.VERSION);

Conclusion

volo and jam losing battle because looks like AMD wouldn't be a first to go module format,
not all application need to on demand loading of modules

ender is more build tool than package manager

From metrics we see that winners are npm and bower, they has
big repos and strong community support. Looks like bowe just filled a gap with asserts management
for browser platform which wasn't included in npm by design.

Component is promising tool but community need some time to learn on how to create and
reusable small components for frontend

jspm gives a good playground to work with es6 standard
